# 数据库结构说明（MySQL）

> 基于 `server/db.ts` 中 `initDb()` 的实际建表语句及数据库实际 schema 整理。

## 1. 数据库概览

- 数据库名：`vinshipping_management`（来自 `DB_NAME`）
- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_0900_ai_ci`
- 存储引擎：`InnoDB`
- 主体业务表：
  - 用户：`users`
  - 包裹：`parcels`
  - 包裹物品：`parcel_items`
  - 包裹状态流转日志：`parcel_status_logs`
  - 包裹状态字典：`parcel_statuses`
  - 订单：`orders`
  - 短信验证码：`otp_codes`
  - 物流商：`logistics_providers`
  - 库位管理：`storage_bins`
  - 单号号段：`number_categories`
  - 单号明细：`tracking_numbers`
  - 地址簿：`address_book`
  - 标签模板：`label_templates`
  - 管理员：`admin_users`
  - 管理员角色：`admin_roles`
  - 管理员角色权限映射：`admin_role_permissions`
  - 管理审计日志：`admin_audit_logs`
  - 航线：`shipping_routes`
  - 航线授权：`shipping_route_grants`
  - 集装箱：`shipping_containers`
  - 班(航)次：`shipping_voyages`
  - 提(运)单：`shipping_bills`

> 说明：主要业务表支持软删除（`deleted_at` 字段），查询默认过滤 `deleted_at IS NULL`。
> 部分表已设置数据库层外键约束（详见各表说明及关系说明）。

## 2.3 `admin_users`（后台管理员表）

| 字段名 | 类型 | 约束/索引 | 说明 |
|---|---|---|---|

| role | VARCHAR(32) | DEFAULT 'admin' | 管理员角色 |
| role_scope | VARCHAR(16) | DEFAULT 'platform', INDEX `idx_admin_role_scope` | 角色作用域（`platform` / `logistics`） |
| role_logistics_provider_id | INT | DEFAULT NULL, INDEX `idx_admin_role_provider` | 角色归属物流商ID（仅 logistics 角色使用） |
| logistics_provider_id | INT | DEFAULT NULL, INDEX `idx_admin_provider` | 管理员归属物流商ID（仅 logistics 管理员使用） |
| status | VARCHAR(32) | DEFAULT 'active' | 管理员状态 |
| last_login | DATETIME | 无 | 最后登录时间（可空） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
| deleted_at | DATETIME | DEFAULT NULL | 软删除时间（NULL=未删除） |

### 角色与状态（接口层约束）

- `status` 合法值：`active`、`disabled`
- `role` 不再限制为固定枚举，需在 `admin_roles` 中存在且通过三元组匹配：
  - `role`
  - `role_scope`
  - `role_logistics_provider_id`
- 当 `role_scope = logistics` 时：
  - `role_logistics_provider_id` 必须有值
  - `logistics_provider_id` 必须有值且需与 `role_logistics_provider_id` 一致
- 当 `role_scope = platform` 时：
  - `role_logistics_provider_id` 应为 `NULL`
  - `logistics_provider_id` 通常为 `NULL`

### 角色权限来源（RBAC）

- 自 `v0.2.24` 起，角色对应权限已从代码内置改为数据库表 `admin_role_permissions` 驱动。
- 登录态返回的 `permissions` 字段来自该表（主路径按 `role_id` 查询，旧数据兼容按 `role` 字符串兜底），用于前后端统一鉴权。
- 若某角色在该表中无记录，系统会回退到默认权限模板（应用层兜底）。

## 2.4 `parcels`（包裹表）

| 字段名 | 类型 | 约束/索引 | 说明 |
|---|---|---|---|
| id | INT | PK, AUTO_INCREMENT | 包裹ID |
| user_id | INT | DEFAULT NULL, INDEX `idx_parcels_user`, FK → `users.id` ON DELETE SET NULL | 所属用户ID（可空，入库时为 NULL） |
| tracking_number | VARCHAR(128) | UNIQUE, NOT NULL, INDEX `idx_parcels_tracking` | 包裹单号（唯一） |
| origin | VARCHAR(255) | NOT NULL | 来源 |
| destination | VARCHAR(255) | NOT NULL | 目的地 |
| weight | DOUBLE | 无 | 重量（可空） |
| length_cm | DOUBLE | 无 | 长度（可空，单位 cm） |
| width_cm | DOUBLE | 无 | 宽度（可空，单位 cm） |
| height_cm | DOUBLE | 无 | 高度（可空，单位 cm） |
| volume | DOUBLE | 无 | 体积（可空） |
| images | TEXT | 无 | 图片URL列表（逗号分隔，可空，支持多张） |
| shelf_location | VARCHAR(64) | DEFAULT NULL | 货架位置（可空） |
| storage_bin | VARCHAR(64) | DEFAULT NULL | 库位号（可空，自由文本） |
| logistics_provider_id | INT | DEFAULT NULL | 绑定物流商ID（可空，逻辑关联 `logistics_providers.id`） |
| status | VARCHAR(64) | DEFAULT 'pending', INDEX `idx_parcels_status` | 货物态（包裹主状态） |
| sub_status | VARCHAR(64) | DEFAULT NULL, INDEX `idx_parcels_sub_status` | 信息态（包裹子状态，可空） |
| status_remark | VARCHAR(255) | DEFAULT NULL | 状态备注（可空，如异常原因） |
| status_updated_at | DATETIME | DEFAULT NULL | 状态最后更新时间 |
| estimated_delivery | DATETIME | 无 | 预计送达时间（可空） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
| deleted_at | DATETIME | DEFAULT NULL | 软删除时间（NULL=未删除） |

### 货物态（主状态，接口层约束）

- `status` 的合法值来自《包裹状态字典》（`parcel_statuses` 表）中 `status_type = '货物态'` 且 `is_enabled = 1` 的 `status_code`，由接口层动态校验（不再使用硬编码枚举）。
- 新入库/新建包裹默认货物态为 `warehoused`（已入库）。
- 历史遗留旧值在启动时（`initDb`）自动迁移为字典编码：

| 旧值 | → 字典编码 | 中文名称 |
|---|---|---|
| arrived | cross_border_arrived | 跨境干线到港 |
| received | warehoused | 已入库 |
| in_transit | cross_border_in_transit | 跨境干线运输中 |
| exception | cargo_exception | 货物异常 |
| pending | awaiting_warehousing | 待入库 |
| pickup_pending | self_pickup_ready | 自提待取 |
| delivered | delivered_signed | 签收完成 |

### 信息态（子状态，接口层约束，可空）

- `sub_status` 的合法值来自《包裹状态字典》中 `status_type = '信息态'` 且 `is_enabled = 1` 的 `status_code`，由接口层动态校验；可空。

### 入库逻辑

- 入库时若 `tracking_number` 已存在（含软删除），更新该记录信息并恢复，不新建。
- 入库时若不存在，新建记录，`user_id` 为 NULL，`status` 为 `warehoused`（已入库），`status_updated_at` 为当前时间。

## 2.11 `admin_role_permissions`（管理员角色权限映射表）

| 字段名 | 类型 | 约束/索引 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键ID |
| role_id | INT | NOT NULL, INDEX `idx_role_id` | 角色ID（主路径，指向 `admin_roles.id`） |
| role | VARCHAR(32) | DEFAULT NULL, INDEX `idx_role` | 旧版兼容字段（按角色编码） |
| permission_code | VARCHAR(64) | NOT NULL, UNIQUE KEY `uk_role_permission_id`(role_id, permission_code) | 权限码（如 `admin.update`） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

### 说明

- 一条记录表示“某角色拥有某权限”。
- 主路径通过 `(role_id, permission_code)` 唯一约束保证不重复授权。
- 保留 `role` 字段用于历史数据兼容，不作为新数据主关联键。
- 当前接口支持角色维度配置：
  - `GET /api/admin/roles`
  - `POST /api/admin/roles`
  - `PUT /api/admin/roles/:code`
  - `DELETE /api/admin/roles/:code`
  - 仅具备对应管理员权限的账号可操作（高敏操作仍需超管约束）。

## 4. 软删除机制

所有主要业务表（`users`、`otp_codes`、`admin_users`、`parcels`、`orders`、`logistics_providers`）均包含 `deleted_at` 字段：
- `NULL` 表示未删除
- 删除操作为 `UPDATE SET deleted_at = NOW()`，不做物理删除
- 所有列表/查询默认过滤 `WHERE deleted_at IS NULL`
- 包裹重新入库时，若已软删除的记录存在则恢复并更新

## 5. 初始化与默认数据规则

- 系统首次启动若 `admin_users` 为空，会创建一个超级管理员。
- 必须提供环境变量：
  - `DEFAULT_ADMIN_USERNAME`
  - `DEFAULT_ADMIN_PASSWORD`（至少 12 位）
  - `DEFAULT_ADMIN_EMAIL`
- 若无 `super_admin`，系统会将最早创建的管理员提升为 `super_admin`。
- 系统首次启动会自动补齐系统角色（`admin`、`super_admin`，`scope=platform`）并写入默认权限种子到 `admin_role_permissions`（若该角色尚无权限记录）。

- 升级前请先做数据库备份（至少包含 `admin_users`、`admin_roles`、`admin_role_permissions` 三表）。
- 建议在低峰期执行升级：
  - 本次包含索引与唯一键调整，执行期间可能短暂影响写入性能。
- 推荐执行顺序：
  1. 先发布后端（含自动迁移逻辑）
  2. 验证表结构与索引已生效
  3. 再发布前端（角色三元组参数）

  1. `admin_roles` 存在 `scope`、`logistics_provider_id` 字段与 `uk_role_scope_provider_code` 唯一键
  2. `admin_role_permissions` 存在 `role_id` 字段与 `uk_role_permission_id` 唯一键

  - 如必须结构回滚，请基于升级前备份在维护窗口执行，避免在线删索引/删列造成业务中断。
